# SeasAGI v5 一ページサマリー

## 1. ワンライナー

SeasAGI は AI パワーユーザーとデベロッパー向けのローカル統合モデルゲートウェイであり、ユーザーがネットワーク出口を管理するのと同じ方法で LLM API 出口を管理できるようにします。

SeasAGI は単なる「キー切り替えツール」ではありません — AI インフラ消費側に「ローカル統合モデルゲートウェイ」のエントリポイントを構築します。

その価値は以下にあります：

- マルチモデル統合の複雑さを「クライアントごとの繰り返し設定」から「一度のローカル設定」に削減
- Key セキュリティに対するユーザーの不安を、体感できるプロダクト信頼に変換
- 実行時の Provider 不安定性を適応型ルーティング劣化に変換し、ユーザーには不可視に
- 使用量のブラックボックスをフル次元のインサイト（モデル/チャネル/タイムライン/コスト/エラー）に変換
- 将来のより高価値な戦略ルーティング、チームエディション、サブスクリプションサービスを、すでに稼働しているローカルゲートウェイ基盤の上に構築

## 2. ユーザーのペインポイント

現在の AI クライアントと IDE プラグインには以下の高頻度ペインポイントがあります：

- 異なるモデルプラットフォームで `Base URL`、`API Key`、モデル名の繰り返し設定が必要
- プラットフォームチャネルとユーザー購入の Key が混在、切り替えコストが高い
- 上流サービスが不安定な際、統一的なフォールバックや診断がない
- ユーザーは自分の Key がプラットフォームに収集または平文保存されることを懸念
- 会話中のモデル切り替えがコンテキストの断絶とハルシネーションを引き起こす
- Provider の頻繁な 429 レート制限、適応型ルーティング優先度調整がない
- どのモデル/チャネルの使用量が最も高く、コストが最適かの直感的インサイトがない

## 3. ソリューション

SeasAGI はローカルに統一された OpenAI 互換エントリポイントを提供し、すべてのサードパーティクライアントはローカルアドレスを一度設定するだけで複数のモデル出口にアクセスできます。

コア機能：

- ローカル統合エントリ：デフォルト `http://127.0.0.1:4318/v1`、リスンポートに応じて動的変更
- デュアルチャネルアーキテクチャ：プラットフォームチャネル + カスタムチャネル
- ユーザー提供の Keys はアップロードされず、ローカル Keychain のみに保存
- ルーティング戦略：`fallback`、`round_robin`、`sticky`
- モデルレベル Combo：明示的な `channel + model` ステップチェーン + ドラッグ＆ドロップ並べ替え + プライマリ/バックアップ/最終手段のロールセマンティクス
- ナビゲーション融合：左ナビを「Combo 最適化」単一エントリに統合、4タブワークベンチ（マイプラン / 最適化提案 / テンプレートセンター / 実行分析）
- Playground デバッグ：Combo セレクタ + 実行チェーン可視化 + ステップフォールバックステータス表示（step_role バッジ）
- デフォルトエントリの Combo 対応：HomePage デフォルト Combo セレクタ + Wails 永続化
- Combo クラウド同期：3層ビュー（ローカル/公式/チーム） + Fetch/Push/Update/Delete 完全同期機能
- Combo メトリクス：リクエスト数 / フォールバック数 / Step1 ヒット率 / 最終ステップヒット率
- スマート最適化 & Combo 協作：マルチステップフォールバック提案、構造差分プレビュー、ワンクリック Combo 適用
- ステップ単位 Provider/Channel 候補プール：各 Step が複数候補をサポート + 自動ランキンク
- リクエストレベル実行制約：max_price / max_latency_ms / data_policy
- Tool-call 専用ルーティング戦略：ツール成功率の高い Provider を優先
- クイック戦略エイリアス：stability-first / cost-first / speed-first / tool-first
- タスクタイプ認識：chat / tools / json / long_context / batch_low_cost 差分化実行
- エンタープライズ BYOK / プラットフォーム共有キャパシティ 2層フォールバック
- セッションスティッキールーティング：最初のユーザーメッセージに基づく SHA1 ハッシュ、30分 TTL、会話中のモデル切り替えを防止
- 動的 429 ペナルティ劣化：PenaltyManager、429 +3、成功 -1、2分自然減衰
- フォールバックソートプリセット：intelligence / speed / budget ワンクリック並べ替え
- 8 Provider Executor：OpenAI、Azure、Anthropic、Gemini、Ollama、DeepSeek、Grok、プラットフォームリレー
- 6 フォーマットトランスレータ：OpenAI Chat/Responses、Anthropic、Gemini、Vertex AI、Passthrough
- Key ヘルスチェック：5分周期プローブ、3連続失敗で自動不健康マーク
- Key レベル Cooldown：Key 単位 120秒 TTL、429 で精密冷却、他の Key に影響なし
- マルチモーダルコンテンツフラット化：テキストのみ配列コンテンツを自動で文字列にマージ
- Playground 即時テスト：チャット UI + モデル選択 + ローカルゲートウェイ通信
- 定数時間 Key 比較：crypto/subtle.ConstantTimeCompare、タイミングサイドチャネルを排除
- Zod フロントエンドバリデーション：8スキーマ + フォーム統合
- ログ＆診断：リンク可視化、CSV エクスポート、構造化エラーコピー
- RTK トークン圧縮：9出力タイプ自動検出 + 圧縮
- フルモーダル API：Embeddings / Image / TTS / STT
- トンネルリモートアクセス：Cloudflare Tunnel + Tailscale Funnel
- マルチアカウントフォールトトレランス：マルチ Key ローテーション + 指数バックオフ + サーキットブレーカー + 7エラールール
- OAuth エコシステム：PKCE + 4 Provider レジストリ + ローカルループバックコールバック + トークン自動リフレッシュ
- ローカルアクセストークン：左ナビ独立ページ、表示 / コピー / リセット
- クラウド機能：登録 / ログイン、プラットフォームチャネル同期、使用量 / 課金照会
- クラウド分析：モデル/チャネル別グループ + コスト推定（GPT-4o 価格） + 直近エラー + リクエストタイムライン + エラー分類分布
- 管理可視化：Recharts BarChart / LineChart / PieChart、6タブ UsagePage
- Caveman 簡潔出力：4スタイル注入
- Reasoning Content：DeepSeek-R1/Kimi/QwQ 推論チェーン注入
- MCP ツール重複除去：12等価マッピング
- 国際化：中国語、英語、日本語、韓国語、システム言語に追従

## 4. なぜ今か

- AI ユーザーが複数のクライアントと複数のモデルを同時に使用することが常態化
- ローカルファースト、セキュリティファースト、統合エントリの需要が急速に上昇
- 「クラウド集約プラットフォーム」は存在するが、「ローカルゲートウェイ + 強固なセキュリティ境界 + 適応型ルーティング + フルチェーン可観測性」を持つ成熟プロダクトは依然として希少
- SeasAGI はまず個人デベロッパー市場に浸透し、その後チームエディションとクロスプラットフォームエディションに拡大可能

## 5. プロダクトポジショニング

- ターゲットユーザー：
  - AI パワーユーザー
  - 個人デベロッパー
  - インディーデベロッパー
  - 小規模 AI チームのテックリード
- 現在の形態：
  - Wails デスクトップクライアント（主要検証プラットフォーム macOS、コード構造は Windows / Linux をカバー）
  - ローカル OpenAI 互換 API ゲートウェイ
- コアシナリオ：
  - 複数モデルプラットフォームへの統合アクセス
  - プラットフォームチャネルと自己購入 Key の切り替え
  - チャネル障害フォールバックと適応型劣化
  - リクエスト障害リンク診断
  - マルチターン会話セッションスティッキネス
  - 使用量インサイトとコスト最適化

## 6. 差別化バリア

### 6.1 明確なセキュリティ境界

- ユーザー提供の Keys はプラットフォームにアップロードされない
- Keys は macOS Keychain に保存
- ローカルアクセストークンはローカル SQLite に保存、クラウドに入らない
- トークン検証は定数時間比較を使用、タイミングサイドチャネル攻撃を排除
- プラットフォームチャネルとユーザーチャネルの境界を UI に明確表示

### 6.2 適応型インテリジェントルーティング

- 単なる「デフォルトチャネル切り替え」ではない
- セッションスティッキネス：マルチターン会話は同じモデルに自動ルーティング、ハルシネーションを防止
- 動的 429 ペナルティ：Provider レート制限で優先度が自動低下、成功で自動回復
- ソートプリセット：intelligence / speed / budget ワンクリック並べ替え
- Key ヘルスチェック：自動プローブ、失敗した Key は自動無効化
- Key 単位 Cooldown：精密制御、他の Key に影響なし
- マルチモーダルコンテンツフラット化：より多くのクライアントフォーマットに互換
- ステップ単位 Provider/Channel 候補プール + 自動ランキンク：同じステップ内で安定性/コスト/レイテンシにより動的に最適選択
- Tool-call 独立ルーティング戦略：ツール成功率でソート、価格優先のみではない
- リクエストレベル動的制約：各コールに max_price / max_latency_ms / data_policy を付与可能

### 6.3 モデルレベル Combo オーケストレーション & ガバナンス

- 障害フォールバックオーケストレーション：静的モデルリストではなく、「プライマリ → バックアップ → 最終手段」の明示的フォールバックチェーン
- ナビゲーション融合：左ナビ単一エントリ「Combo 最適化」、ワークベンチ統合フロー：設定→提案→適用→検証
- スマート最適化 & Combo 協作：マルチステップフォールバック提案 + 構造差分プレビュー + ワンクリック適用
- クラウド同期：3層ビュー（ローカル/公式/チーム） + 完全同期（Fetch/Push/Update/Delete）
- 実行分析：Combo ヒット率 / フォールバック率 / 平均試行回数ダッシュボード
- サーバーコントロールプレーン：3レベル Combo 管理 + バージョン公開/ロールバック + 差分化ロールアウト
- エンタープライズガバナンス：可視化ポリシー（ロール/プラン/ユーザーグループ/テナント 4レベル） + デプロイ承認フロー
- Combo ランタイム計装：リクエスト数 / フォールバック数 / Step1 ヒット率 / 最終ステップヒット率
- クイック戦略エイリアス：stability-first / cost-first / speed-first / tool-first
- タスクタイプ認識：chat/tools/json/long_context による差分化実行

### 6.4 フルチェーン可観測性

- ログにヒットチェーンとステップ単位の失敗理由を含む
- エラー分類分布（429/401/403/404/timeout/500/503）
- リクエストタイムラインチャート（24時間/7日/30日）
- モデル/チャネル別グループ分析
- 直近エラークイックビュー
- コスト推定（GPT-4o 価格ベースライン）
- 管理チャート可視化（BarChart / LineChart / PieChart）

### 6.5 ローカルゲートウェイ体験

- すべてのクライアントはローカルアドレスを一度設定するだけ
- Chatbox、Cherry Studio、Cursor などで設定を繰り返し変更する必要がない
- Playground 即時テスト、チャネル設定後すぐに検証

### 6.6 拡張可能なアーキテクチャ

- 外部は統一的に OpenAI 互換
- 内部はプロトコル正規化 + Provider Executor でより多くの上流を拡張
- Executor レジストリが新しい Provider の動的登録をサポート
- OAuth PKCE フローが任意の OAuth 2.0 Provider をサポート
- チームエディション、ポリシーロールアウト、設定同期へスムーズに拡張可能

## 7. 3サブプロジェクト構成

- `SeasAGI-Client/`：クライアントサブプロジェクト、内部に `src-app` 元構造を保持、Wails デスクトップクライアントメインラインソースとクライアントビルドスクリプトを搭載
- `SeasAGI-Server/`：サーバーサブプロジェクト、内部に `platform-api`、`relay-gateway`、`src-admin`、`deploy` 元構造を保持、コントロールプレーン、データプレーン、管理ダッシュボードとデプロイスクリプトを搭載
- `web-docs/`：ウェブサイト & ドキュメントサブプロジェクト、内部に `src-web` を保持、メインラインアーキテクチャ/PRD/スケジュールドキュメント、および `cc-switch`、`9router` 等の歴史または参照ディレクトリ
- ルートディレクトリ：トップの `README.md`、`SeasAGI.v5.md` とワンクリックビルドスクリプト `build-all.sh` のみを保持
